## System overview

AmoraDB is built around five core components that work together to provide efficient JSONL-based data storage:

- **AmoraDB**: Main database class that manages tables and metadata
- **Table**: Handles data operations, batching, and compaction
- **Query**: Processes queries with index optimization
- **IndexManager**: Maintains hash and sorted indices for fast lookups
- **SimpleCache**: LRU cache for frequently accessed records

<Info>
All components extend Node.js `EventEmitter`, allowing you to listen to database events in real-time.
</Info>

## Component interaction

Here's how the components work together during common operations:

### Insert flow

When you insert a record, the data flows through multiple components:

```javascript
const record = await db.table('users').insert({ name: 'Alice' });
```

1. **Table.insert()** (Table.js:297-321) adds metadata fields:
   - `_id`: UUID generated by `uuidv4()`
   - `_created`: ISO timestamp
   - `_modified`: ISO timestamp

2. **SimpleCache** stores the record for fast access (Table.js:308)

3. **IndexManager** updates all indices for this table (Table.js:309)

4. Record is added to `pendingWrites` array (Table.js:313)

5. When `pendingWrites.length >= batchSize` (default 1000), records are flushed to disk (Table.js:315-316)

6. **EventEmitter** fires an `insert` event (Table.js:319)

### Query flow

Queries leverage indices when possible for optimal performance:

```javascript
const users = await db.table('users')
  .find({ age: { $gte: 18 } })
  .sort('name', 'asc')
  .limit(10)
  .execute();
```

1. **Query.where()** analyzes conditions to identify indexable fields (Query.js:19-35)

2. Conditions are split into:
   - `indexableConditions`: Can use indices (Query.js:39-59)
   - `nonIndexableConditions`: Require full scan (Query.js:16)

3. **Query.getIndexCandidateSet()** retrieves matching IDs from indices (Query.js:142-185)

4. **Table.loadRecordById()** loads records not in memory (Query.js:254)

5. Non-indexable conditions filter results (Query.js:219-221)

6. Results are sorted, skipped, and limited (Query.js:223-233)

<Note>
If a table has more records than `cacheSize`, Query automatically streams from the JSONL file instead of loading everything into memory (Query.js:197-199).
</Note>

### Update flow

Updates use pending operations to batch writes efficiently:

```javascript
const updated = await db.table('users').update(id, { status: 'active' });
```

1. **Table.get()** retrieves the current record (Table.js:377)

2. Record is updated with new fields plus `_modified` timestamp (Table.js:383-389)

3. Record is added to `pendingUpdates` Map (Table.js:391)

4. **SimpleCache** is updated (Table.js:392)

5. **IndexManager.updateIndices()** updates relevant indices (Table.js:393)

6. **Table.queueSave()** schedules compaction if needed (Table.js:394)

7. `update` event is emitted (Table.js:395)

## Data flow to disk

AmoraDB uses a multi-stage write strategy for reliability and performance:

### Batched writes

New records are batched before writing to disk (Table.js:270-285):

```javascript
this.pendingWrites = []; // Array of new records
this.batchSize = 1000;   // Flush threshold
```

When the batch is full, records are appended to the JSONL file:

```javascript
const lines = this.pendingWrites.map(record => JSON.stringify(record) + '\n');
await fs.appendFile(this.filePath, lines.join(''));
```

### Compaction

Updates and deletes create "dead" records in the JSONL file. Compaction rewrites the file to remove them.

Compaction triggers when (Table.js:182-188):

- Pending operations exceed 30% of total records (`compactionThreshold = 0.3`)
- OR more than 10,000 pending operations AND 60+ seconds since last compaction

```javascript
needsCompaction() {
  const totalOps = this.pendingUpdates.size + this.pendingDeletes.size + this.deletedIds.size;
  const timeSinceCompaction = Date.now() - this.lastCompaction;
  
  return (totalOps > this.metadata.recordCount * this.compactionThreshold) ||
         (totalOps > 10000 && timeSinceCompaction > this.compactionInterval);
}
```

### Atomic operations

Compaction uses atomic file operations to prevent data loss (Table.js:197-246):

1. Write updated data to `{table}.jsonl.tmp`
2. Read existing JSONL file line-by-line
3. Skip deleted records, use updated versions from `pendingUpdates`
4. Close temp file
5. Atomically rename temp file to replace original: `fs.rename(tempPath, filePath)`

<Warning>
If compaction fails, the temporary file is deleted and the original remains intact (Table.js:264-266).
</Warning>

## File structure

AmoraDB organizes files in a simple directory structure:

```
data/
└── mydb/
    ├── _metadata.json        # Database metadata
    ├── users.jsonl           # User records (one per line)
    ├── users.meta.json       # Table metadata and indices
    ├── posts.jsonl
    └── posts.meta.json
```

### Database metadata

Stored in `_metadata.json` (AmoraDB.js:21-26):

```json
{
  "version": "1.1.0",
  "created": "2024-03-01T12:00:00.000Z",
  "modified": "2024-03-04T15:30:00.000Z",
  "tables": {
    "users": {
      "created": "2024-03-01T12:00:00.000Z",
      "modified": "2024-03-04T15:30:00.000Z",
      "indices": ["email", "age"],
      "schema": {}
    }
  }
}
```

### Table metadata

Stored in `{table}.meta.json` (Table.js:31-36):

```json
{
  "autoIncrement": 1,
  "indices": ["email", "age"],
  "recordCount": 1523,
  "modified": "2024-03-04T15:30:00.000Z"
}
```

## EventEmitter integration

Both `AmoraDB` and `Table` extend `EventEmitter` to provide real-time notifications:

```javascript
class AmoraDB extends EventEmitter { /* ... */ }
class Table extends EventEmitter { /* ... */ }
```

You can listen to events at the database or table level:

```javascript
// Database-level events
db.on('ready', () => console.log('Database ready'));
db.on('error', (err) => console.error('Database error:', err));
db.on('close', () => console.log('Database closed'));

// Table-level events
const users = db.table('users');
users.on('insert', (record) => console.log('Inserted:', record._id));
users.on('update', (record) => console.log('Updated:', record._id));
users.on('delete', (record) => console.log('Deleted:', record._id));
```

<Info>
See the [Events](/concepts/events) page for a complete list of available events.
</Info>

## Initialization sequence

When you call `db.init()`, several steps occur (AmoraDB.js:30-46):

1. **ensureDataDirectory()**: Creates `data/{dbname}` directory
2. **loadMetadata()**: Reads `_metadata.json` or creates it
3. **migrateFromJSON()**: Migrates old `.json` files to `.jsonl` format
4. **loadTables()**: Discovers and loads all `.jsonl` files
5. For each table:
   - **loadMetadata()**: Reads table metadata (Table.js:61-74)
   - If `recordCount <= cacheSize`: **loadAllRecords()** into memory (Table.js:46-47)
   - Otherwise: **buildIndicesFromFile()** only (Table.js:49)
6. Emit `ready` event (AmoraDB.js:39)

<Note>
Small tables (under `cacheSize` records) are fully loaded into memory for maximum performance. Large tables stream from disk as needed.
</Note>

## Configuration options

You can configure AmoraDB behavior when creating an instance (AmoraDB.js:14-20):

```javascript
const db = new AmoraDB('mydb', {
  dataPath: './data',        // Where to store files
  autoSave: true,            // Auto-save on changes
  cacheSize: 1000,          // Max records in cache
  indexAutoCreate: true,     // Auto-create indices
  compression: false         // Enable compression (future)
});
```

These options affect how components interact and when data is written to disk.
